# 📑 Project Instructions: Vecontab

## 📌 Contexto do Projeto

- **Nome:** Vecontab (V-E-C-O-N-T-A-B)
- **Empresa:** VEC Sistemas (SaaS Proprietário - Copyright 2026)
- **Objetivo:** ERP Contábil/Fiscal (Manutenção de empresas, controle de tributos/obrigações e integração com APIs financeiras).
- **Modelo de Negócio:** Multitenancy (Isolamento por Schema no Postgres).
- **Tenant Principal:** VEC Sistemas (Não aparece como cliente; gerencia integrações e usuários SUPER).

## 🛠 Stack Tecnológica Oficial

- **SO/Ambiente:** Fedora Linux 43 (`dnf5` para pacotes).
- **Backend:** Go (Golang) - Foco em performance absoluta (Benchmark > 4.5x Python).
- **Frontend:** React + Next.js 16 + TypeScript + PrimeReact/PrimeFlex + React Query.
- **Banco de Dados:** PostgreSQL 18.3 (Driver `pgx/v5`).
- **Arquitetura:** Clean Architecture, SOLID e Domain-Driven Design (DDD).

## 🚀 Comandos e Caminhos (Fedora 43)

- **Backend:** `go run cmd/api/main.go` (em `vecontab/backend`)
- **Frontend:** `npm run dev` (em `vecontab/frontend`)
- **Banco Local:** `psql -h localhost -U camaral -d vecontab`
- **Migrations:** SEMPRE em `vecontab/migrations` (Execução via SQL puro).
- **Binários (build local):** sempre em `bin/` dentro do módulo (ex.: `agente-local/bin/vecx-agent.exe`). A pasta `bin/` está no `.gitignore` da raiz do repositório.

## ⚖️ Diretrizes Obrigatórias (Backend/Banco)

1. **Sem ORM:** Proibido o uso de ORMs. Acesso via SQL explícito + `pgx` para otimização total.
2. **Identificadores:** IDs de entidades devem ser sempre **UUID**.
3. **Erros:** Tratar erros com contexto para log, mas sem expor dados sensíveis em produção.
4. **Camadas:** Separação rígida entre Models, Handlers, Use Cases e Repositories.
5. **Schemas e Tenants** Sempre que mencionado, a nova tabela deverá ter a estrutura criada em no schema template_tenant e irradiada para os tenants existentes. Se for mencionado como 'public', então entra no schema public.

## 🎨 Padrões de Interface (Frontend)

1. **Consultas:** Preferir `useQuery` em vez de `useEffect` para fetch de dados.
2. **Layout de Botões:**
   - Botão **"Atualizar"**: Ícone fixo no canto inferior da página (nunca botão textual superior).
   - Botões de linha em Tables/TreeTables: Seguir o modelo adotado no módulo "Municípios".
3. **Checkbox (Padrão Aprovado):**
   - Usar wrapper `.field-checkbox` (PrimeFlex) + `<input type="checkbox">` nativo + `<label>`.
   - Evitar o componente `Checkbox` do PrimeReact para prevenir artefatos visuais ("fantasmas").
4. **TabView dentro de Dialog (Padrão Clientes):**
   - **CSS:** Ocultar ink bar (`pt={{ inkbar: { style: { display: 'none' } } }}`).
   - **Layout:** Não usar `justify-content: space-between` no `ul`.
   - **Altura:** `min-height` fixa via CSS (ex: `min(58vh, 38rem)`); evitar `ResizeObserver`.
   - **Atalhos:** Manter atalhos numéricos (1, 2, 3) à direita via `position: absolute`.

## 👤 Perfis de Usuário e Segurança

- **SUPER:** Dono (VEC Sistemas). Acesso total, incluindo configurações globais de API.
- **ADMIN:** Administrador do escritório contábil (Gerencia usuários do seu tenant).
- **USER:** Usuários operacionais com poderes restritos por permissão.
- **Restrição de Acesso:** Abas e botões devem ser desabilitados (`disabled`) conforme o perfil (Ex: Certificado Digital apenas ADMIN/SUPER).

## 📝 Regras de Escrita e Linguagem

1. **Ortografia:** O "ç" **nunca** é utilizado antes de "e" ou "i". Usar "c" simples para manter o som de "s" (ex: "vencimento", não "vençimento"). O mesmo vale para acentos em identificadores técnicos.
2. **Domínio:** Considerar o calendário brasileiro e feriados fiscais (postergação para dia útil).

## 🔒 Protocolo de Escopo do Agente

1. **Escopo travado por padrão:** Alterar somente os arquivos diretamente solicitados pelo usuário.
2. **Sem efeito colateral:** Não alterar módulos fora do pedido sem autorização explícita do usuário.
3. **Pré-confirmação objetiva:** Antes de editar, informar em 1 linha quais arquivos serão tocados.
4. **Pós-relato obrigatório:** Após editar, listar os arquivos alterados e o motivo de cada alteração.
5. **Bloqueio por risco de impacto:** Se surgir necessidade de mexer fora do escopo, parar e pedir confirmação antes de continuar.
6. **Mudanças emergenciais fora de escopo:** Só podem ocorrer quando forem indispensáveis para o pedido e com confirmação explícita do usuário.
